# STITCH_ANALYSIS

## Что находится в экспорте

Экспорт из Google Stitch в папке `design/stitch/` содержит 3 файла:

- `screen.png` — high-fidelity скрин главного экрана.
- `code.html` — HTML-прототип экрана с Tailwind-конфигом, цветовыми токенами, типографикой, структурой секций и Telegram WebApp-инициализацией.
- `DESIGN.md` — текстовое описание дизайн-системы, визуальных принципов, токенов, компонентных правил и product tone.

Важно: ZIP-архива как такового в исходной папке не было. В `C:\Future\Минус 40\Stitch\1` уже лежал распакованный экспорт, и он был скопирован в `design/stitch/`.

## Что является source of truth

Для дальнейшей интеграции я бы использовал такой приоритет:

1. `DESIGN.md`
   Это главный source of truth для принципов интерфейса, цветовой системы, типографики, spacing, shape language и правил компонентов.

2. `screen.png`
   Это главный source of truth для визуальной композиции экрана, иерархии блоков, плотности интерфейса и общего mobile layout.

3. `code.html`
   Это технический reference-слой: он полезен как черновой blueprint структуры экрана, токенов и Tailwind-классов, но не должен переноситься в проект буквально как production source.

## Какие экраны есть

В текущем Stitch-экспорте есть один экран:

- `Минус 40 - Главная`

По содержимому `screen.png` и `code.html` это главный dashboard-экран веса с фиксированной кнопкой действия и нижней навигацией.

Отдельных экранов истории, настроек, деталей записи или формы добавления веса в экспорт не входит.

## Вывод по интеграции

Stitch-экспорт достаточно полный, чтобы считать его визуальным источником правды для следующей итерации главного экрана.

Для реализации не нужно копировать `code.html` напрямую. Правильнее:

- взять `DESIGN.md` как дизайн-систему
- взять `screen.png` как композиционный эталон
- использовать `code.html` как reference для структуры секций, токенов и UI-поведения

## Качество экспорта

Качество экспорта можно оценить как хорошее для MVP-этапа.

Плюсы:

- есть high-fidelity экран;
- есть отдельный `DESIGN.md` с описанием визуальной системы;
- есть экспортированный HTML с layout-структурой и токенами;
- есть явная связь с Telegram Mini App контекстом;
- есть usable design tokens: цвета, типографика, spacing, border radius.

Ограничения:

- в экспорте только один экран;
- нет отдельных локальных иконок и image assets;
- нет production-ready React/TypeScript кода;
- нет явной разбивки на компонентную библиотеку;
- нет отдельных состояний экранов, кроме одного обзорного состояния.

## Пригодность для дальнейшей разработки

Экспорт пригоден для разработки как дизайн-референс, но не как прямой источник production-кода.

Что будем использовать:

- `DESIGN.md` как главный документ по дизайн-системе;
- `screen.png` как визуальный ориентир;
- `code.html` как reference по секциям, safe-area логике и приоритетам блоков;
- цветовые и типографические токены;
- структуру главного экрана и приоритет CTA.

Что не будем использовать напрямую:

- сам `code.html` как основу финальной вёрстки;
- CDN Tailwind-конфиг из экспорта как production-решение;
- внешний HTML как есть без адаптации к текущей архитектуре React/Vite;
- части интерфейса, не подтверждённые roadmap текущего этапа.

## Одна следующая задача разработки

Следующая задача:

Реализовать вычисляемые блоки данных для Stitch-главного экрана без полной смены вёрстки:

- `изменение за неделю`
- `прогресс до цели 110 кг`
- `ближайшие цели 140 / 130 / 120 / 110`

Причина:

Именно эти данные являются функциональным фундаментом Stitch-дизайна. После их появления можно будет уверенно переносить сам layout, не подменяя макет заглушками и не переписывая экран дважды.
