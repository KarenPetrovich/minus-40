# SKILLS_POLICY.md

## Назначение

Этот документ определяет, какие skills используются в проекте, когда их вызывать и как решать конфликты между ними.

## Главный принцип

Skills должны помогать текущему этапу проекта, а не тянуть проект в сторону лишней сложности.

Если skill не усиливает текущую задачу, он не должен становиться активным только потому, что “может пригодиться”.

## Структура проекта для skills

```text
Skills/
├── active/
├── standby/
└── future/
```

Смысл папок:

- `active/` — skills, которые реально поддерживают текущий этап;
- `standby/` — skills, полезные, но не обязательные на каждом шаге;
- `future/` — skills для следующих этапов roadmap.

## Активные skills

### Telegram Mini App Specialist

Статус: active

Когда вызывать:

- при проектировании экранов Telegram Mini App;
- при работе с Telegram Web App API;
- при обсуждении ограничений Mini App среды;
- при принятии решений по UX внутри Telegram.

### Spec-Driven Development

Статус: active

Когда вызывать:

- перед новой функциональной итерацией;
- когда нужно быстро зафиксировать границы следующей задачи;
- когда есть риск “сразу уйти в код”.

### Frontend UI Engineering

Статус: active

Когда вызывать:

- при разработке экранов и компонентов;
- при принятии решений по mobile UI;
- при улучшении читаемости и структуры интерфейса.

### Playwright Interactive / Browser Testing

Статус: active

Когда вызывать:

- при проверке пользовательских сценариев;
- при отладке интерфейса;
- при визуальной проверке графика, форм и экранов.

## Standby skills

### Security Threat Model

Статус: standby

Когда вызывать:

- перед появлением backend;
- перед хранением чувствительных данных;
- перед интеграциями, которые расширяют поверхность риска.

### Data Privacy Compliance

Статус: standby

Когда вызывать:

- при проектировании хранения данных;
- при добавлении фото еды, активности и другой чувствительной информации;
- при принятии решений по retention и приватности.

### Accessibility

Статус: standby

Когда вызывать:

- перед крупными UI-итерациями;
- при проверке форм, контрастов, читаемости и управления;
- перед стабилизацией пользовательского интерфейса.

### Security Best Practices

Статус: standby

Когда вызывать:

- при появлении серверной логики;
- при работе с хранением, API и пользовательскими данными;
- при secure-by-default ревью кода.

## Приоритет при конфликте мнений

Если разные skills дают разные рекомендации, приоритет такой:

1. Прямые инструкции пользователя.
2. `DECISION_LOG.md`.
3. `PROJECT_KNOWLEDGE.md`.
4. `CURRENT_PHASE.md`.
5. Активные domain/platform skills текущего этапа.
6. Активные engineering skills текущего этапа.
7. Standby skills.
8. Future skills.

Дополнительное правило:

Если skill предлагает решение, которое расширяет проект за пределы текущего этапа roadmap, такое решение не принимается без отдельного явного решения.

## Что делать при конфликте “продукт против технологии”

Если технологический skill предлагает сложное решение, а продуктовые документы требуют простого MVP:

- побеждает продуктовый фокус;
- берётся минимальное техническое решение, достаточное для текущего этапа.

## Что делать при конфликте “качество против скорости”

На старте проекта:

- побеждает простое, но чистое решение;
- не принимаются ни хрупкие хаки, ни преждевременная инженерная тяжесть.

Ниже — полезные кандидаты, которые пока лучше держать как явную опцию:

### Data Visualization

Источник:

- `antvis/chart-visualization-skills`

Рекомендуемая команда:

```bash
npx skills add antvis/chart-visualization-skills
```

Рекомендуемое проектное расположение:

- `Skills/future/chart-visualization`

### Weight Loss Skill

Источник:

- `NanoRhino/weight-loss-skill`

Причина неавтоматической установки:

- репозиторий не дал корректно установить root skill через helper в текущем формате.

Рекомендуемое проектное расположение:

- `Skills/future/weight-loss-skill`

### Computer Vision

Источник:

- `roboflow/computer-vision-skills`

Рекомендуемое проектное расположение:

- `Skills/future/computer-vision`

### AI Engineer / Data Analyst / Backend Engineer

Источник:

- `VoltAgent/awesome-claude-code-subagents`

Примечание:

- это скорее subagents/personas, чем классические SKILL.md-пакеты;
- их лучше подключать осознанно, когда проект дойдёт до соответствующего этапа.
