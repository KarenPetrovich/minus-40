# AGENTS_PLAN.md

## Цель документа

Зафиксировать, каких публичных специалистов стоит использовать в проекте «Минус 40», в каком порядке и на каких этапах.

Важно: этот документ не означает автоматическую установку. Он фиксирует решение о выборе.

## Принципы отбора

- Берём только специалистов, полезных именно для Telegram Mini App по снижению веса.
- Не берём нерелевантные маркетинговые, SEO, growth и крипто-направления.
- На старте предпочитаем небольшой, понятный набор.
- Сначала нужны специалисты, которые помогают сделать MVP Этапа 1.

## Выбранные агенты для старта

### 1. Telegram Mini App Specialist

Зачем нужен:

- помогает не ошибиться в специфике Telegram Mini App;
- удерживает фокус на мобильном сценарии внутри Telegram;
- полезен при проектировании UX, жизненного цикла Mini App и интеграции с Telegram Web App API.

Риск:

- некоторые публичные Telegram-скиллы часто перегружены нерелевантными сценариями вроде монетизации, TON и платежей;
- при использовании нужно жёстко ограничивать его рамками нашего продукта.

### 2. Spec-Driven Development

Зачем нужен:

- помогает не перепрыгнуть сразу в код;
- фиксирует маленькие и понятные спеки под каждую итерацию;
- хорошо сочетается с подходом «сначала MVP».

Риск:

- если использовать слишком формально, может замедлить маленькие задачи.

### 3. Frontend UI Engineering

Зачем нужен:

- усиливает качество интерфейса;
- помогает держать мобильный UI аккуратным, понятным и не «AI-looking»;
- полезен для React/TypeScript интерфейсов.

Риск:

- может тянуть проект в сторону избыточной полировки раньше времени.

### 4. Playwright / Browser Testing

Зачем нужен:

- помогает быстро проверять реальные пользовательские сценарии;
- полезен для регрессионной проверки интерфейса и форм;
- особенно важен для Mini App, где визуальные и runtime-проблемы легко пропустить.

Риск:

- можно преждевременно построить тяжёлую тестовую инфраструктуру до стабилизации MVP.

### 5. Data Visualization

Зачем нужен:

- график веса — ядро первой версии продукта;
- помогает выбрать честную и читаемую визуализацию без лишнего шума.

Риск:

- если увлечься визуализацией слишком рано, можно потратить время на графики вместо базового сценария ввода веса.

### 6. Security Threat Model

Зачем нужен:

- помогает заранее увидеть риски по персональным данным и каналам интеграции;
- полезен ещё до появления полноценного backend.

Риск:

- на ранней стадии может дать слишком много «бумажной» безопасности, если запускать его слишком часто.

### 7. Data Privacy Compliance

Зачем нужен:

- вес, фото еды и активность — чувствительные пользовательские данные;
- помогает проектировать хранение и обработку аккуратнее.

Риск:

- для персонального MVP легко перегрузить проект формальными требованиями раньше времени.

## Резервные агенты

Эти специалисты полезны, но не обязательны в самый первый день.

### Accessibility

Нужен для:

- проверки контрастности, читабельности, доступности форм и управления.

Почему резерв:

- важен, но на старте может работать как периодический аудит, а не как постоянно активный специалист.

### Security Best Practices

Нужен для:

- secure-by-default решений в TypeScript/JavaScript и будущих серверных частях.

Почему резерв:

- до появления backend и auth достаточно threat model + здравых практик.

### Product Management / Product Skills

Нужен для:

- декомпозиции гипотез и контроля за тем, чтобы не раздувать продукт.

Почему резерв:

- часть этой функции уже покрывает Spec-Driven Development и текущая документация проекта.

## Будущие агенты по этапам развития

### Этап анализа еды

- Computer Vision
- AI Engineer
- Nutrition Specialist

Назначение:

- разбор фото еды;
- извлечение признаков;
- классификация;
- прикладные рекомендации по питанию.

### Этап аналитики

- Data Analyst
- Data Visualization

Назначение:

- поиск закономерностей в весе, еде и активности;
- объяснимые графики и отчёты;
- выявление факторов снижения и откатов.

### Этап Telegram-интеграции

- Telegram Mini App Specialist
- Backend Engineer

Назначение:

- развитие Mini App логики;
- интеграция API;
- хранение данных;
- синхронизация и возможные уведомления.

### Этап поведенческой поддержки

- Behaviour Change Specialist
- Weight Loss Coach

Назначение:

- поддержка дисциплины;
- работа с ночной едой, срывами и перееданием;
- мягкие, но действенные сценарии возвращения в трек.
